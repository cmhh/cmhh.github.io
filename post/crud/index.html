Creating a Basic Read-only Service

Overview

Data services are great. Among other use cases, they let others access your data remotely in a relatively user-friendly way, and avoid the need to download a whole database or large flat file just to find a handful of relevant records (and do this again after the next database update). And having published data (and procedures) in this way, it is also then only too easy to build appealing single page applications (SPAs) on top.

However, it can be pretty easy to over-engineer things. There are a number of specifications and formats to select from, like GraphQL or OData. There are plenty of frameworks, like Spring Boot, ASP.NET Core, or Play. And having made one or two services, it can be easy to think investing in a generic solution that can address all your future data service needs might be a worthwhile venture.

But if your use case is simply that you have a moderately sized database that you want to make available in a strictly read-only fashion, then a simple, bespoke solution may suffice. Put another way, if you have a simple database and it’s mostly used to retrieve data using minor variations on a few simple queries, you might not get huge milage out of something that is beautifully engineered for general purpose use.

Having said all this, let’s build a simple data service. The idea will be to build a service with one or two simple end-points, and without using a framework as such. We’ll make two versions: one using Node.js, and one using Akka HTTP. Node.js is wildly popular, with an absolutely huge library of middleware available, so one really can put together something useful with surprisingly little code. Similarly, Akka HTTP can be used to put together a data service with ‘just enough’ code. I like Scala as a language, and I use it a fair bit for general purpose programming. Akka HTTP lets me take a library made with Scala, and then expose it as a web service to non-Scala users. Creating a Database

But Do I Really Need a Database?

You need a database of some kind. A CSV file itself is a database of sorts. But do you need a full-blown RDBMS such as MySQL? Or a NoSQL option such as MongoDB? The answer is, it depends. If you have a lot of data that needs to be persistent, highly available, and / or you want some control over who can access it, then databases offer all of this functionality and more. But if your needs are simpler, then something more basic, perhaps bespoke, might be perfectly plausible. For example, an in-memory associative array might work perfectly well in the same sort of settings as you might use a key-value store like Redis (which is also a great database option) if your data fits in memory, and can be recreated quickly from source in the case of failure.
